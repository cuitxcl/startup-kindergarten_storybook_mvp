# 后端领域架构重构方案

## 当前问题

`server/src/controllers/api.rs` 已经承担了过多职责：路由注册、请求解析、权限校验、领域流程、mock 分支、数据库分支、审计日志、导出下载、公开链接、运营后台和 readiness 检查都集中在一个文件中。当前文件超过 5000 行，继续新增功能会带来三个问题：

- 修改一个领域时容易影响其他领域。
- 权限、错误处理、审计日志等横切逻辑难以复用。
- Loco 的 controller/service/repository 分层优势没有发挥出来。

后端下一步不应继续在 `api.rs` 中追加业务，而应先做领域化拆分。

## 目标架构

第一阶段保持现有 API 路径和返回结构不变，只调整代码组织。

## 分层职责

重构后建议采用 `controller -> application -> domain/service -> repository` 的方向，而不是让 controller 直接编排所有业务。

| 层 | 负责内容 | 不负责内容 |
| --- | --- | --- |
| Controller | 路由、请求体解析、Path/Query/Header 提取、HTTP 状态码、返回 envelope | 复杂业务编排、跨表事务、领域规则 |
| Application | 一个用户用例的完整编排，例如“创建普通绘本”“批量生成定制绘本”“复制市场模板”“确认投稿隐私” | SQL 细节、HTTP 细节、底层 provider 调用细节 |
| Domain / Service | 领域规则、状态机、生成策略、隐私检测、可导出规则、权限语义 | 请求解析、数据库 CRUD |
| Repository | 数据库读写、查询条件、事务内持久化 | 业务流程决策 |
| Worker / Task | 异步执行、重试、任务状态推进 | 同步 HTTP 入口决策 |

`application` 层的价值是把“一个按钮背后的完整业务动作”收束到明确用例里。否则 controller 拆成多个文件后，复杂流程仍然会散落在各个 handler 中，只是从一个大文件变成多个中等文件。

## 领域边界

| 领域 | 负责内容 | 不负责内容 |
| --- | --- | --- |
| Auth | 登录、注册、当前用户、邀请接受、token 解析 | workspace 内权限策略 |
| Workspaces | 空间列表、空间上下文、dashboard 汇总 | 成员管理细节 |
| Organization | 园所成员、班级、审计日志 | 儿童档案内容 |
| Children | 儿童档案 CRUD、归档恢复、老师班级可见范围 | 定制绘本生成 |
| Parent Intakes | 家长资料收集链接、公开提交、确认入档 | 分享绘本 |
| Storybooks | 普通绘本、定制绘本、页面、角色、复制、批量定制派生 | AI 任务执行 |
| Generation | 生成任务、重试、取消、恢复、图片下载 | 具体绘本 CRUD |
| Delivery | 导出任务、分享链接、公开导出下载 | 市场投稿 |
| Marketplace | 市场模板列表、详情、复制 | 园所投稿审核流 |
| Submissions | 园所投稿、隐私确认 | 平台运营审核 |
| Operator | 平台审核、成本、存储、readiness | 园所日常工作流 |
| Public Links | 不登录访问的分享链接、家长资料链接解析 | 登录态空间能力 |

## 横切能力抽取

这些逻辑不应继续散落在各 controller 中：

- `auth_context`：解析 bearer token、当前用户、dev token。
- `workspace_guard`：`workspace member / editor / admin / operator` 权限。
- `classroom_scope`：`school_teacher` 的班级儿童范围。
- `api_error_map`：数据库错误到业务错误的转换。
- `audit_events`：审计事件名称和 payload 构造。
- `delivery_guard`：可导出、可分享、隐私风险拦截。
- `pagination`：分页参数、分页响应。
- `readiness`：试用环境检查，与业务 controller 分离。

建议目录：

```text
server/src/domains/common/
  auth_context.rs
  workspace_guard.rs
  classroom_scope.rs
  error_map.rs
  audit_events.rs
  pagination.rs
```

## 迁移顺序

### 第 1 步：路由聚合拆分

新增 `controllers/routes.rs`，由它聚合各领域 routes：

```rust
pub fn routes() -> Routes {
    Routes::new()
        .merge(health::routes())
        .merge(auth::routes())
        .merge(workspaces::routes())
        .merge(children::routes())
        .merge(storybooks::routes())
}
```

验收标准：`cargo run -- routes` 中路径不变。

### 第 2 步：抽出 application 用例入口

先不要把所有逻辑塞进新的 controller 文件。每迁移一个领域时同步创建对应 application 模块：

```text
server/src/application/storybooks/
  mod.rs
  create_plain.rs
  derive_custom.rs
  derive_custom_batch.rs
  update_page.rs
```

controller 只负责调用 application：

```rust
let output = storybooks::derive_custom_batch::execute(&ctx, actor, workspace_id, storybook_id, input).await?;
Ok((StatusCode::CREATED, Json(Envelope::new(output))))
```

验收标准：新 controller 文件不再直接包含大量 repository 调用、审计 payload 拼装和跨领域权限判断。

### 第 3 步：先拆低耦合 controller

优先迁移：

- `health`
- `auth`
- `workspaces`
- `organization`
- `children`

这些领域对生成、导出、市场的依赖较少，适合先拆。

### 第 4 步：拆故事与生成领域

迁移：

- `storybooks`
- `generation`
- `delivery`

这里需要先抽出 `workspace_guard`、`classroom_scope`、`delivery_guard`，否则只是把大文件拆成多个相互复制的小文件。

### 第 5 步：拆市场、投稿、运营后台

迁移：

- `marketplace`
- `submissions`
- `operator`
- `public_links`

这一步完成后，旧 `controllers/api.rs` 应删除，或只保留兼容入口。

## 阶段一冻结原则

- 不改 URL。
- 不改 envelope：成功 `{ data }`，失败 `{ error }`。
- 不改数据库表。
- 不重写 repository。
- 不新增生产化能力。
- 每迁移一个领域就跑一次 `cargo fmt && cargo test`。

## 优先拆分清单

| 优先级 | 文件/模块 | 原因 |
| --- | --- | --- |
| P0 | `controllers/routes.rs` | 先建立领域路由入口，后续迁移不影响 App 注册 |
| P0 | `application/mod.rs` | 建立用例层，避免新 controller 继续变胖 |
| P0 | `domains/common/workspace_guard.rs` | 权限校验是所有 workspace API 的共同依赖 |
| P0 | `controllers/health.rs` | 最低风险，用于验证拆分方式 |
| P0 | `controllers/auth.rs` | 登录注册相对独立 |
| P1 | `controllers/children.rs` | 真实试用高频，且边界清晰 |
| P1 | `controllers/storybooks.rs` | 产品核心，但体量较大，放在 common 抽取之后 |
| P1 | `controllers/generation.rs` | 与 worker/provider 强相关，应独立 |
| P2 | `controllers/operator.rs` | 运营后台体量大，但不影响老师主路径 |

## 建议下一步

下一轮代码实施从“最小安全拆分”开始：

1. 新增 `controllers/routes.rs` 和 `controllers/health.rs`。
2. 新增 `application/mod.rs`，先建立 application 层目录，不急着迁移复杂用例。
3. `app.rs` 从 `api::routes()` 改为 `routes::routes()`。
4. 把 `/api/health` 从 `api.rs` 移走。
5. 跑 `cargo fmt`、`cargo test`、`cargo run -- routes`。

这一步很小，但能证明新的 Loco 领域 controller 结构可行。之后再按领域逐个迁移，不再继续扩大 `api.rs`。
