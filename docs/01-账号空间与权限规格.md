# 账号空间与权限规格

## 1. 设计目标

本规格定义系统中的账号、空间、空间成员、空间角色、组织边界和权限规则。

它服务于后续所有功能：

- 普通绘本生成。
- 定制绘本生成。
- 个人孩子资料。
- 园所儿童档案。
- 园所老师邀请。
- 绘本市场复制。
- 园所作品投稿市场。
- 家长资料提交链接。
- 绘本分享链接。

核心设计原则：

- 账号不直接决定所有权限，权限由“当前空间 + 空间角色 + 资源归属”共同决定。
- 所有核心业务资源必须有清晰归属，不能只依赖前端状态判断权限。
- 默认拒绝访问，只有明确授权的角色和资源范围才允许操作。
- 个人空间和园所空间必须隔离。
- 链接访问只能访问链接授权范围内的数据，不能升级成空间成员权限。

## 2. 核心概念

### 2.1 User 账号

账号代表一个可登录的人。

规则：

- 每个注册账号默认创建一个个人空间。
- 同一个账号可以加入多个园所空间。
- 同一个账号在个人空间中是 `personal_owner`，在园所空间中可以是 `school_teacher` 或 `school_admin`。
- 账号本身只表示身份认证结果，不直接表示业务权限。

### 2.2 Workspace 空间

空间是数据归属和权限边界。

空间类型：

| 类型 | 说明 |
| --- | --- |
| `personal` | 个人空间，每个注册账号默认拥有一个 |
| `school` | 园所空间，用于老师、班级、园所绘本和管理 |
| `platform` | 平台后台空间，可作为后续运营后台边界 |

说明：

- `link_visitor` 不拥有 workspace，它通过 token 访问指定资源。
- 市场模板是平台级资源，不属于个人或园所空间；用户使用时必须复制到自己的当前空间。

### 2.3 Workspace Member 空间成员

空间成员表示某个账号在某个空间中的身份和角色。

示例：

```text
user A
├── personal workspace: personal_owner
└── school workspace: school_teacher
```

规则：

- 权限判断必须基于 active 的 `workspace_member`。
- 同一用户在同一空间第一版只允许一个主角色。
- 后续如果需要一个用户在同一园所既是老师又是管理员，可通过角色升级为 `school_admin`，不建议同时存多条 active 成员记录。

### 2.4 Current Workspace 当前空间

当前空间是用户正在操作的数据边界。

规则：

- 登录后必须有明确当前空间。
- 如果用户只有个人空间，默认进入个人空间。
- 如果用户有多个空间，默认进入上次使用空间；如果无记录，优先个人空间。
- 所有创建操作默认写入当前空间。
- 所有列表查询默认只返回当前空间内可见数据，除非查询的是市场公开资源。

### 2.5 Link Visitor 链接访问者

链接访问者是不一定登录的外部访问者。

第一版场景：

- 家长通过资料提交链接填写孩子资料。
- 家长或外部用户通过分享链接查看或下载绘本。

规则：

- 链接访问依赖带有效期的 token。
- token 只授予指定动作和指定资源范围。
- 链接访问者不是空间成员。
- 链接访问者不能进入个人空间或园所空间。

## 3. Actor Model

| Actor | 身份来源 | 生效范围 | 核心能力 |
| --- | --- | --- | --- |
| `anonymous` | 未登录、无有效链接 | 公开页面 | 浏览公开页面，不能访问受保护资源 |
| `link_visitor` | 有效链接 token | 链接指定资源 | 提交资料、查看分享绘本 |
| `personal_owner` | 登录账号 + 个人空间 member | 个人空间 | 个人绘本、个人孩子资料、个人导出 |
| `school_teacher` | 登录账号 + 园所空间 member | 指定园所空间 | 园所绘本、授权班级儿童、导出交付 |
| `school_admin` | 登录账号 + 园所空间 admin member | 指定园所空间 | 园所设置、成员、班级、用量、市场投稿 |
| `platform_operator` | 登录账号 + 平台后台权限 | 平台后台 | 市场模板、投稿审核、平台运营，后续完善 |

## 4. Authentication Assumptions

第一版认证假设：

- 用户通过邮箱或手机号注册登录。
- 登录后获得 session token 或等价登录态。
- 每个 authenticated request 都能识别 `user_id`。
- 对 workspace-scoped API，客户端必须携带当前 `workspace_id`，或服务端从会话中读取当前空间。
- 服务端必须验证当前账号是否是该 workspace 的 active member。

链接访问认证：

- 家长资料提交链接和绘本分享链接使用不可猜测 token。
- token 必须存储哈希或等价安全形式。
- token 应有状态、过期时间和访问范围。
- 链接访问不要求登录，但不能获得空间成员权限。

## 5. Protected Resources

受保护资源按归属分为四类。

### 5.1 User-scoped

- 用户基础资料。
- 登录凭证。
- 个人默认空间。
- 用户空间成员列表。

### 5.2 Workspace-scoped

- 普通绘本。
- 定制绘本。
- 绘本页面。
- 绘本角色与道具设定。
- 图片资产。
- 导出记录。
- 分享链接。
- 个人孩子资料。
- 园所儿童档案。
- 班级。
- 园所成员。
- 园所用量记录。

### 5.3 Platform-scoped

- 市场模板。
- 平台精选内容。
- 园所投稿审核记录。
- 平台运营配置。

### 5.4 Link-scoped

- 家长资料提交链接。
- 绘本分享链接。
- 链接访问日志。

## 7. Ownership And Scope Rules

### 7.1 默认拒绝

除以下资源外，所有业务数据默认需要认证和授权：

- 公开市场模板的公开字段。
- 有效链接 token 授权的资料提交页。
- 有效分享 token 授权的绘本查看页。

### 7.2 个人空间规则

- 每个账号默认有且只有一个 active personal workspace。
- personal workspace 的 owner 是创建该账号的用户。
- 个人空间资源只能由该空间的 `personal_owner` 访问。
- 个人空间孩子资料不能被园所空间直接引用。
- 如果用户要把个人内容用于园所，应通过复制/导入动作产生园所空间副本，不能共享原记录。

### 7.3 园所空间规则

- 园所空间内资源归属于该 school workspace。
- `school_teacher` 只能访问自己所在园所空间。
- `school_teacher` 对儿童档案的访问受班级授权限制；后端 children API 会按 `workspace_members.classroom_ids` 过滤列表、详情、编辑和定制绘本选用的儿童。
- 家长提交待确认队列包含未归档儿童资料，第一版仅 `school_admin` 可查看和确认入档；`school_teacher` 不直接访问全园待确认提交。
- `school_admin` 可管理该园所空间内成员、班级、内容和基础用量。
- 园所空间不能访问其他园所空间数据。

### 7.4 市场规则

- 市场模板是 platform-scoped 资源。
- 普通用户和老师只能复制市场模板，不能直接修改市场原件。
- 复制市场模板时，系统在当前空间创建普通绘本副本。
- 副本与市场模板保留来源关系，但后续编辑不影响原模板。

### 7.5 园所投稿规则

- 第一版只允许园所普通绘本投稿。
- 定制绘本禁止投稿市场。
- 含儿童照片、具体儿童隐私、家长敏感信息的内容禁止投稿。
- 投稿必须经过平台审核后才能公开上架。
- 未通过审核的投稿只能被提交园所和平台审核人员查看。

### 7.6 链接访问规则

- 家长资料提交链接只允许创建一条或多条指定范围内的待确认提交。
- 分享链接只允许查看或下载指定绘本。
- 链接 token 不允许调用空间成员 API。
- 链接 token 不允许列出 workspace 资源。
- 链接 token 过期、撤回或禁用后必须立即拒绝访问。

## 8. Tenancy Model Summary

最高层客户边界是 Workspace。

```text
User
├── personal workspace
└── school workspace membership(s)

Workspace
├── personal
└── school
```

租户设计结论：

- 用 workspace 统一承载个人空间和园所空间。
- 用 workspace_member 表达用户在空间中的角色。
- 用 workspace_id 作为绝大多数业务资源的隔离字段。
- 平台市场资源不归属于某个客户空间，但使用时必须复制进 workspace。
- 链接访问不创建租户成员关系，只创建 token-scoped 授权。

## 9. Core Scopes

| Scope | 包含资源 | 访问方式 |
| --- | --- | --- |
| User scope | 账号资料、登录凭证、用户拥有的空间列表 | 登录用户本人 |
| Personal workspace scope | 个人绘本、个人孩子资料、个人导出 | `personal_owner` |
| School workspace scope | 园所绘本、班级、儿童档案、成员、用量 | `school_teacher` / `school_admin` |
| Platform scope | 市场模板、投稿审核、平台运营 | `platform_operator` |
| Link scope | 资料提交、分享查看 | token 授权访问 |

## 10. Membership And Role Model

### 10.1 空间成员状态

| 状态 | 含义 |
| --- | --- |
| `active` | 正常成员，可访问空间 |
| `invited` | 已邀请但未接受，不可访问空间 |
| `suspended` | 被停用，不可访问空间 |
| `left` | 已离开空间，不可访问空间 |
| `removed` | 被管理员移除，不可访问空间 |

第一版可只实际使用 `active`，邀请状态由 invitation 表表达；但成员表应预留状态字段。

### 10.2 角色

| role | workspace_type | 说明 |
| --- | --- | --- |
| `personal_owner` | `personal` | 个人空间拥有者 |
| `school_teacher` | `school` | 园所老师 |
| `school_admin` | `school` | 园所管理员 |
| `platform_operator` | `platform` | 平台运营 |

约束：

- `personal_owner` 只能出现在 personal workspace。
- `school_teacher` 和 `school_admin` 只能出现在 school workspace。
- `platform_operator` 只能出现在 platform workspace 或单独平台权限模型中。

### 10.3 老师邀请流程

第一版只支持园所管理员邀请老师。

```text
school_admin
-> 创建邀请
-> 填写邮箱或手机号
-> 可选绑定班级授权
-> 发送邀请链接/邀请码
-> 用户打开链接
-> 注册或登录
-> 查看邀请信息
-> 接受邀请
-> 创建或激活 workspace_member(role=school_teacher)
```

邀请状态：

| 状态 | 可接受 | 说明 |
| --- | --- | --- |
| `pending` | 是 | 等待接受 |
| `accepted` | 否 | 已接受 |
| `expired` | 否 | 已过期 |
| `revoked` | 否 | 管理员撤回 |

规则：

- 只有该 school workspace 的 `school_admin` 可创建邀请。
- 邀请只绑定一个 school workspace。
- 邀请可选绑定一个或多个 classroom。
- 接受邀请必须登录；未注册用户先注册再接受。
- 接受成功后不影响用户个人空间。
- 同一用户可以接受多个园所邀请，成为多个园所空间成员。

| 资源 | 归属字段 | 说明 |
| --- | --- | --- |
| 普通绘本 | `workspace_id` | 属于个人或园所空间 |
| 定制绘本 | `workspace_id`, `source_storybook_id` | 属于当前空间，来源普通绘本不可被覆盖 |
| 绘本页面 | `storybook_id` | 通过 storybook 继承 workspace 归属 |
| 绘本角色/道具 | `storybook_id` | 通过 storybook 继承 workspace 归属 |
| 个人孩子资料 | `workspace_id`, `owner_user_id` | personal workspace 内使用 |
| 园所儿童档案 | `workspace_id`, `classroom_id` | school workspace 内使用 |
| 班级 | `workspace_id` | 只属于 school workspace |
| 导出记录 | `workspace_id`, `storybook_id` | 继承绘本归属 |
| 分享链接 | `workspace_id`, `storybook_id` | token scoped |
| 家长资料提交链接 | `workspace_id`, `classroom_id` | token scoped |
| 市场模板 | platform-scoped | 用户只能复制 |
| 园所投稿 | `workspace_id`, `source_storybook_id` | 审核通过后生成或关联市场模板 |

## 12. Isolation And Sharing Rules

### 12.1 空间隔离

- 查询 workspace-scoped 资源时必须过滤 `workspace_id`。
- 任何写操作都必须验证当前用户是目标 workspace 的 active member。
- personal workspace 和 school workspace 之间不能直接共享同一业务记录。
- 跨空间复用必须通过复制产生新记录。

### 12.2 园所内可见性

普通绘本建议支持可见性字段：

| visibility | 说明 |
| --- | --- |
| `private` | 仅创建者和 school_admin 可见 |
| `workspace` | 同园所授权成员可见 |
| `market_submission` | 正在投稿审核 |
| `market_listed` | 已作为市场模板公开，或与市场模板关联 |

第一版可简化：

- 个人空间绘本默认仅本人可见。
- 园所老师草稿默认仅创建者和 `school_admin` 可见。
- 园所管理员可将普通绘本提交市场审核。

### 12.3 分享链接

分享链接只授权具体绘本的只读查看或下载。

分享链接不可：

- 列出空间内其他绘本。
- 查看孩子资料。
- 编辑绘本。
- 访问园所成员、班级或用量。

## 13. Enforcement Points

权限必须在以下位置执行：

### 13.1 API 层

- 所有 workspace-scoped API 必须验证当前用户在目标 workspace 的 active membership。
- 所有 class-scoped API 必须验证 teacher 是否拥有班级访问权限。
- 所有市场管理 API 必须验证 `platform_operator`。
- 所有投稿审核 API 必须验证 `platform_operator`。
- 所有链接访问 API 必须验证 token 状态、过期时间和资源范围。

### 13.2 数据访问层

- 查询条件必须包含 `workspace_id` 或通过父资源追溯 workspace。
- 不允许先查出资源再只在前端过滤。
- 删除建议使用归档状态，避免误删影响审计和来源追踪。

### 13.3 UI 层

UI 可以隐藏无权限操作，但不能作为唯一权限控制。

UI 必须明确展示：

- 当前空间。
- 当前空间角色。
- 当前内容归属个人空间还是园所空间。
- 市场模板复制后将进入哪个空间。
- 投稿市场前的隐私和脱敏提醒。

## 14. Failure Behavior

| 场景 | 行为 |
| --- | --- |
| 未登录访问受保护 API | 返回未认证错误，提示登录 |
| 登录用户访问非成员空间 | 返回无权限或资源不存在 |
| 成员状态非 active | 返回无权限 |
| teacher 访问未授权班级 | 返回无权限 |
| link token 无效 | 返回链接无效 |
| link token 过期 | 返回链接已过期 |
| invitation 过期 | 提示联系园所管理员重新邀请 |
| invitation 已接受 | 提示已加入园所或进入园所空间 |
| 投稿包含禁止内容 | 阻止提交或进入人工审核失败状态 |

建议：

- 对跨空间资源访问，优先返回“资源不存在或无权限”，避免泄露其他空间资源存在性。
- 对邀请和链接场景，可以返回更明确的用户提示，因为用户已持有链接。

### 15.5 Constraints And Validation

关键约束：

- 所有 workspace-scoped 表必须有 `workspace_id`，或可通过父资源追溯到 workspace。
- `workspace_members.role` 必须匹配 `workspaces.type`。
- `classrooms.workspace_id` 必须是 school workspace。
- `child_profiles.profile_scope=personal` 时 workspace 必须是 personal。
- `child_profiles.profile_scope=school` 时 workspace 必须是 school。
- `storybooks.storybook_type=custom` 时必须有来源普通绘本和目标孩子资料。
- 定制绘本不能投稿市场。
- market submission 的 source storybook 必须是 plain storybook。
- share link 和 intake link token 必须可撤回、可过期。

### 15.6 Indexes And Query Patterns

建议索引：

- `users(email)` unique where email not null
- `users(phone)` unique where phone not null
- `workspaces(type, status)`
- `workspace_members(user_id, status)`
- `workspace_members(workspace_id, role, status)`
- `workspace_members(workspace_id, user_id)` unique for active member
- `workspace_invitations(token_hash)` unique
- `workspace_invitations(workspace_id, status, expires_at)`
- `classrooms(workspace_id, status)`
- `classroom_memberships(workspace_id, user_id, status)`
- `child_profiles(workspace_id, profile_scope, status)`
- `child_profiles(workspace_id, classroom_id, status)`
- `storybooks(workspace_id, storybook_type, status)`
- `storybooks(workspace_id, visibility, updated_at)`
- `exports(workspace_id, storybook_id, status)`
- `share_links(token_hash)` unique
- `parent_intake_links(token_hash)` unique
- `marketplace_templates(status, age_group, use_scene)`
- `marketplace_submissions(workspace_id, status)`

### 15.7 Lifecycle Rules

账号：

- 用户删除建议先软删除或停用。
- 停用用户后，其 active membership 不应继续允许访问。

空间：

- personal workspace 不建议删除，用户注销时统一归档。
- school workspace 可归档；归档后不允许新增内容或邀请成员。

成员：

- 移除成员不删除其历史创作记录。
- 成员被移除后不可访问空间，但其创建的绘本仍属于空间。

邀请：

- pending 邀请过期后不可接受。
- revoked 邀请不可接受。
- accepted 邀请保留审计记录。

绘本：

- 删除建议用 `archived`，保留来源关系和导出记录。
- 市场模板来源的绘本副本可自由编辑，不反向影响模板。

投稿：

- submitted 后可由平台审核。
- approved 可生成 marketplace_template。
- rejected 保留原因。
- listed/unlisted 控制市场可见性。

链接：

- 过期或撤回后不可访问。
- 分享链接不应删除原访问审计。

### 15.8 Migration Notes

如果当前系统已有 `schools/teachers/classrooms/children/storybooks` 等表，迁移时建议：

- 新增 `workspaces`，将现有 school 迁移为 `type=school` workspace。
- 新增 `workspace_members`，将现有 teachers 转成 school workspace member。
- 为现有 storybooks、children、assets、exports 补 `workspace_id`。
- 将个人用户相关数据迁移到 personal workspace。
- 对市场模板单独建表，不复用用户普通绘本原表作为公开模板源。

## 16. API And UI Implications

### 16.1 API 设计原则

- 所有业务 API 应显式处于当前 workspace 上下文。
- 创建资源时服务端根据当前 workspace 写入 `workspace_id`。
- 市场 API 是 platform/public scope，不依赖当前 workspace 浏览，但复制模板时必须指定目标 workspace。
- 链接访问 API 使用 link token，不使用 workspace session。

推荐 API 分组：

- `/api/me`
- `/api/workspaces`
- `/api/workspaces/:workspace_id/switch`
- `/api/workspaces/:workspace_id/members`
- `/api/workspaces/:workspace_id/invitations`
- `/api/workspaces/:workspace_id/classrooms`
- `/api/workspaces/:workspace_id/children`
- `/api/workspaces/:workspace_id/storybooks`
- `/api/marketplace/templates`
- `/api/marketplace/submissions`
- `/api/link/intakes/:token`
- `/api/link/share/:token`

### 16.2 UI 设计原则

所有登录后的主界面必须显示：

- 当前空间名称。
- 当前空间类型。
- 当前用户角色。
- 空间切换入口。

危险或跨边界操作必须提示：

- 将市场模板复制到哪个空间。
- 园所作品投稿后会进入平台审核。
- 投稿前必须确认不包含儿童隐私。
- 分享链接会让外部访问者看到哪些内容。

## 17. Cross-layer Check

数据库能支撑权限：

- workspace_members 表能判断用户是否属于空间以及角色。
- workspace_id 能支撑空间隔离查询。
- classroom_memberships 能支撑老师班级授权。
- token 表能支撑 link scoped 访问。
- marketplace_submissions 能支撑园所投稿审核。

API 能支撑 UI：

- 当前空间、空间角色、成员列表、邀请状态都可查询。
- 市场模板可浏览，复制时可生成当前空间普通绘本副本。
- 链接访问可独立于登录态完成资料提交或绘本查看。

主要不一致风险：

- 如果部分资源没有 `workspace_id` 或父资源链路，权限会变得不可检查。
- 如果 UI 只显示“老师/普通用户”而不显示当前空间，用户容易误操作。
- 如果市场投稿直接引用原园所绘本公开展示，后续编辑和隐私风险会变高。

## 18. Open Security Risks

- 儿童资料属于敏感信息，后续需要更细的访问审计和脱敏策略。
- 市场投稿需要人工审核或自动检测，否则可能泄露儿童或园所隐私。
- 分享链接和资料提交链接必须防止暴力猜测、长期有效和越权访问。
- 平台运营后台后续需要独立审计，不应拥有无限制静默查看能力。
- 老师离职后的历史内容归属需要明确：内容保留在园所，账号失去访问权。

## 19. 验收标准

- 注册账号后自动拥有 personal workspace，并在其中拥有 `personal_owner`。
- 登录用户只能访问自己 active membership 对应的 workspace。
- `school_admin` 可以创建老师邀请。
- 用户接受有效邀请后，在对应 school workspace 成为 `school_teacher`。
- 过期或撤回邀请不能接受。
- `school_teacher` 不能查看其他园所空间。
- `school_teacher` 不能查看未授权班级儿童档案。
- 个人空间孩子资料不能被园所空间直接引用。
- 市场模板复制后生成当前空间普通绘本副本，编辑副本不影响市场模板。
- 定制绘本不能提交市场。
- 链接访问者只能访问 token 指定资源。

## 20. 待确认项

- 第一版是否允许普通用户自助创建园所空间并成为 `school_admin`，还是只允许平台创建园所。
- 第一版 `school_teacher` 默认不能创建班级；班级创建由 `school_admin` 完成，老师只能在授权班级范围内创建或维护儿童档案。
- 园所普通绘本从老师草稿变成园所可见内容，是否需要 `school_admin` 发布动作。
- 分享链接第一版是否支持下载 PDF，还是只支持在线查看。
- 平台运营是否独立使用 platform workspace，还是单独后台权限模型。
