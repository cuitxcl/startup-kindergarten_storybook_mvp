# 异步任务与生成队列设计

## Async workload summary

Kindleaf 的长耗时能力分为三类：AI 文本生成、AI 插图生成、PDF 导出。它们都不应该阻塞 HTTP 请求；接口只负责校验权限、创建任务、返回任务状态，真正执行交给 worker。

当前阶段仍使用 mock 执行器和 mock AI provider，但任务语义按真实异步系统设计：先写入 `queued` 任务，再进入 `running`，最后落到 `succeeded` 或 `failed`。为了不影响前端原型，mock 执行器会在同一个请求内立即完成任务；后续接真实 worker 时保持 API 返回结构不变。

生成能力已在后端抽出 provider 边界：

- `KINDLEAF_GENERATION_PROVIDER=mock`：默认值，返回结构化 mock 结果。
- `KINDLEAF_GENERATION_PROVIDER=deepseek`：启用 DeepSeek 文本 HTTP provider，覆盖 `storybook_plan`、`storybook_roles`、`storybook_pages`、`customization_plan`。
- `KINDLEAF_GENERATION_PROVIDER=seedream`：启用字节跳动 Seedream 图片 provider，覆盖 `storybook_page_image`。
- 不显式设置 provider 且同时配置 `DEEPSEEK_API_KEY` 与 `SEEDREAM_API_KEY` 时，会启用组合 provider：文本走 DeepSeek，插图走 Seedream。
- 真实 provider 已有 fake smoke 验收入口；生产化仍需继续补限流、成本、重试退避、内容安全和对象存储策略。

## Job types

| 任务类型 | 用途 | 载荷引用 | 结果 |
|---|---|---|---|
| `storybook_plan` | 根据教学目标生成故事方案 | `workspace_id`、可选 `storybook_id`、需求摘要 | 故事大纲、主题、页数建议 |
| `storybook_roles` | 生成主角、老师形象和重复配角 | `storybook_id`、角色约束 | 角色设定、外观、故事作用 |
| `storybook_pages` | 生成分页文字和插图提示词 | `storybook_id`、页数、方案版本 | 分页标题、正文、插图 prompt |
| `storybook_role_reference_image` | 生成或编辑单个角色参考图 | `storybook_id`、`role_id`、prompt、可选参考图 | 角色参考图 URL，并写回角色 |
| `storybook_page_image` | 生成或重绘单页插图 | `storybook_id`、`page_id`、prompt、`reference_role_ids`、`reference_image_urls` | 图片 URL、生成摘要 |
| `customization_plan` | 为某个儿童生成定制改写方案 | `storybook_id`、`child_id`、定制强度 | 个性化替换点、注意事项 |
| `export_pdf` | 导出绘本 PDF | `storybook_id`、导出配置 | PDF 文件 URL |

任务 payload 只放稳定引用和必要参数，不放儿童完整敏感档案快照。worker 执行时按权限和数据边界重新读取资源。

当前 mock provider 的 `GenerationJob.output_json` 使用统一结构：

- `schema_version`：当前为 `generation.mock.v1`。
- `provider`：当前为 `mock`，后续可替换为真实 AI provider 名称。
- `mode`：对应任务类型。
- `message`：可直接展示给前端的任务摘要。
- 任务专属结果：例如 `plan`、`roles`、`pages`、`image`、`customization`。

## Text provider contract

### API summary

后端内部通过 `AiGenerationProvider` 调用生成能力。HTTP API 和任务表不直接依赖具体模型；provider 负责把 `GenerationRequest` 转为模型请求，再把模型响应规整为 `GenerationJob.output_json`。

### Resources

- `GenerationRequest`：文本生成请求，包含 `job_type` 和 `input`。
- `ImageGenerationRequest`：图片生成请求，包含 `page_id` 和 `prompt`。
- `GenerationProviderError`：provider 层错误，最终映射为 API 错误 envelope。
- `ConfiguredGenerationProvider`：按 `KINDLEAF_GENERATION_PROVIDER` 选择具体实现。

### Endpoints

provider 不是公开 REST endpoint。公开入口仍保持：

- `POST /api/workspaces/:workspace_id/generation-jobs`
- `POST /api/workspaces/:workspace_id/storybooks/:storybook_id/pages/:page_id/image-tasks`
- `GET /api/workspaces/:workspace_id/generation-jobs`
- `GET /api/workspaces/:workspace_id/generation-jobs/:job_id`

### Request payloads

`GenerationRequest`：

```json
{
  "job_type": "storybook_plan",
  "input": {
    "title": "一起试试看",
    "theme": "学习轮流",
    "age_group": "4-5 岁"
  }
}
```

`ImageGenerationRequest`：

```json
{
  "target_id": "page-or-role-uuid",
  "target_type": "page",
  "mode": "storybook_page_image",
  "prompt": "温暖幼儿园教室，老师和孩子围坐阅读",
  "image_mode": "reference_image",
  "reference_images": [
    {
      "url": "/generated-images/seedream-role.png",
      "source": "storybook_role",
      "role_id": "role-uuid",
      "label": "小兔米米"
    }
  ],
  "edit_instruction": "保持角色不变，只把背景改成午睡区",
  "strength": 0.55
}
```

### Response payloads

所有 provider 必须返回统一 envelope 内的 `output_json` 结构：

```json
{
  "schema_version": "generation.provider.v1",
  "provider": "deepseek",
  "mode": "storybook_plan",
  "message": "绘本方案已生成",
  "plan": {
    "title": "一起试试看",
    "theme": "学习轮流",
    "age_group": "4-5 岁",
    "summary": "string",
    "page_count": 6,
    "outline": [
      { "page_range": "1", "goal": "进入场景", "beat": "string" }
    ],
    "role_requirements": ["string"],
    "review_points": ["string"]
  }
}
```

mock provider 当前使用 `generation.mock.v1`，字段结构与真实 provider 保持同形，方便前端和后续 worker 复用。

### Validation rules

- `job_type` 允许文本任务：`storybook_plan`、`storybook_roles`、`storybook_pages`、`customization_plan`；图片任务：`storybook_role_reference_image`、`storybook_page_image`。
- 图片任务可由 mock provider 或 Seedream 图片 provider 执行；生产化前仍需补图片审核、对象存储和访问生命周期。
- `deepseek` provider 必须配置 `DEEPSEEK_API_KEY`，否则返回 provider 错误。
- `deepseek` provider 会调用 `DEEPSEEK_BASE_URL/chat/completions`，请求体包含 `response_format: {"type": "json_object"}`。
- 发送给模型的 user prompt 必须包含 JSON 输出要求和目标 `response_schema`，避免模型返回 Markdown 或散文文本。
- provider 输出必须是可解析 JSON，且包含 `schema_version/provider/mode/message`。

### Auth and permissions

provider 层不做用户权限判断。权限仍由 controller/repository 在入队前完成：

- 用户必须属于当前 workspace。
- 只有 owner/admin/teacher 可以创建生成任务。
- teacher 的儿童定制任务必须受班级授权限制。

### Error model

- provider 配置缺失：返回 `GenerationProviderError`，API 层映射为 `state_conflict`。
- 不支持的任务类型：返回 `GenerationProviderError`。
- 外部模型超时、限流或响应不可解析：后续真实 HTTP 调用接入后应写入任务失败状态和错误摘要。

当前生成任务的失败面分两层：

- 请求前校验失败：例如未登录、无空间权限、任务类型不支持、绘本或页面不存在。这类错误不创建任务，直接返回 API error envelope。
- provider 执行失败：例如 DeepSeek 未配置密钥、网络超时、限流、模型返回不可解析 JSON。这类错误会保留已创建的任务，并把 `generation_jobs.status` 更新为 `failed`，`output_json` 写入：

```json
{
  "schema_version": "generation.error.v1",
  "provider": "deepseek",
  "mode": "storybook_plan",
  "message": "生成任务失败，可稍后重试",
  "error": {
    "code": "provider_failed",
    "message": "安全截断后的错误摘要",
    "retryable": true
  }
}
```

前端拿到 `status=failed` 时应显示失败原因摘要和“重新生成”入口。当前已提供重试入口：`POST /api/workspaces/:workspace_id/generation-jobs/:job_id/retry`。接口只接受 `failed` 任务，复用原 `input_json` 和原 job id，执行后更新为 `succeeded` 或再次 `failed`，并维护 `attempt_count`、`last_error`、`next_run_at`。

用户发现排队任务不需要继续时，可以调用取消入口：`POST /api/workspaces/:workspace_id/generation-jobs/:job_id/cancel`。当前只允许取消 `queued` 或 `failed` 待重试任务；`running/succeeded` 返回 `409 state_conflict`，避免和 worker 执行过程产生并发冲突。取消后同一 job 进入 `canceled`，清空 `next_run_at/locked_by/locked_at`，并写入 `generation.canceled.v1` 输出摘要。

### Open questions or versioning risks

- 真实 provider 的 `schema_version` 使用 `generation.provider.v1`，mock 当前仍保留 `generation.mock.v1` 以便测试区分。
- DeepSeek 文本 provider 和 Seedream 图片 provider 已接入 HTTP 调用，但默认 smoke 和本地演示仍使用 `mock`，避免测试依赖外部网络和密钥；真实 provider 通过 fake smoke 单独验收。
- 真实模型输出需要 JSON schema 校验，否则可能出现字段缺失、类型错误或幼儿不适龄内容。

## Enqueue and scheduling rules

- 老师、园所管理员和个人空间所有者可以在自己有权限的空间内创建生成任务。
- `storybook_page_image` 必须绑定存在于该空间内的绘本和页面。
- `export_pdf` 必须绑定存在于该空间内的绘本；公开分享链接导出时通过 token 解析绘本。
- 同一绘本内的 `storybook_plan -> storybook_roles -> storybook_pages -> storybook_page_image` 是产品流程顺序，后端状态机应拒绝明显越级的真实执行；mock 阶段只保证任务类型合法。
- 全班批量定制后续应拆成一个批量父任务和多个儿童子任务，避免单个大任务失败导致整班不可恢复。

## Retry and failure model

- 生命周期：`queued -> running -> succeeded / failed / canceled`。
- `queued` 表示请求已被接受但尚未执行。
- `running` 表示 worker 已领取任务；领取时写入 `locked_by` 和 `locked_at`，用于并发保护和卡住任务排查。
- `succeeded` 必须写入可展示的 `output_json`，导出任务还应写入 `file_url`。
- `failed` 必须写入错误摘要，当前同时写入 `output_json.error` 和 `last_error`；可重试错误会写入 `next_run_at`，重试时 `attempt_count` 递增。
- `canceled` 表示用户或管理员主动停止任务；当前只允许从 `queued/failed` 进入，避免取消正在执行或已写回结果的任务。
- 可重试错误：AI 服务超时、限流、对象存储临时失败、PDF 渲染临时失败。
- 不重试错误：权限失效、资源不存在、输入校验失败、内容安全拒绝。
- 重试必须幂等：同一 job 重放不能重复创建多个正式绘本、多个上架模板或多个不可区分的导出文件。

## Progress and result visibility

- 前端通过 `GET /api/workspaces/:workspace_id/generation-jobs` 展示当前空间最近生成任务，通过 `GET /api/workspaces/:workspace_id/generation-jobs/:job_id` 查询单个任务。
- PDF 导出目前创建接口直接返回 `ExportJob`；后续可补 `GET /api/workspaces/:workspace_id/storybooks/:storybook_id/exports/:job_id`。
- 用户可见状态文案：
  - `queued`：已加入生成队列
  - `running`：正在生成
  - `succeeded`：已完成
  - `failed`：生成失败，可重试
  - `canceled`：已取消
- 工作台和绘本详情应优先展示当前绘本最近一个未完成任务，让老师知道下一步是等待、重试还是继续编辑。
- 失败任务的“重新生成”不需要前端重新拼装原始需求；直接调用 retry 接口即可，后端复用已保存的 `input_json`。

## Concurrency and idempotency notes

- 同一页面的插图重绘可以允许多次，但页面最终只采用最新成功任务的结果。
- 同一绘本同一阶段的文本生成后续应增加幂等键，例如 `storybook_id + job_type + generation_stage_version`。
- PDF 导出可以重复生成，但同一绘本同一版本应优先复用最近成功文件。
- worker 更新状态时必须带当前状态 guard，例如只允许 `queued -> running`，只允许 `running -> succeeded/failed`。
- worker 领取任务时使用 `for update skip locked`，只领取 `queued` 或到达 `next_run_at` 的 `failed` 任务；已加锁超过 15 分钟的任务允许被重新领取。
- 公开分享导出不带用户身份，审计日志 `actor_user_id` 为空，仍应记录 `resource_id` 和 `storybook_id` 摘要。

## Operational controls

- 园所管理员可在审计日志看到分享、导出、投稿等关键动作。
- 平台运营后续需要任务列表：按状态、任务类型、空间、创建时间过滤。
- 后续生产化应增加：
  - 死信队列或 `failed` 任务重放入口。
  - 单空间并发限制，避免一个园所占满生成资源。
  - 任务耗时、失败原因、重试次数指标。
  - 敏感 payload 脱敏日志，避免儿童信息进入外部日志。

## Open risks

- 当前 `generation_jobs` 表已有 `attempt_count`、`last_error`、`next_run_at`、`locked_by`、`locked_at`；真实 worker 前仍需要实现独立 worker 进程、超时扫描和死信队列。
- 当前 mock 执行器在请求内立即完成，不能证明真实耗时生成下的长轮询和前端进度条体验。
- 图片已有 Seedream provider 边界和本地 PNG 写入能力，但接对象存储前不能作为生产分享资源。
- PDF 结果已经是本地基础 PDF 文件，支持封面页和逐页文字内容；专业排版、字体嵌入、插图排版和对象存储仍未生产化。
- 全班批量定制需要父子任务模型，目前只完成单个儿童定制基础。
