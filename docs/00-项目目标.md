# Kindleaf 项目目标

## 1. 一句话目标

Kindleaf 要成为面向幼儿园老师和园所的 AI 绘本生产与内容资产管理系统，帮助老师把教学目标快速转化为普通绘本，再基于儿童档案派生定制绘本，并让园所把优秀内容沉淀、复用、分享和投稿到市场。

## 2. 项目愿景

幼儿园老师每天都有大量真实教育场景：规则引导、情绪安抚、习惯培养、主题活动、家园沟通、个别儿童支持。很多场景都适合用绘本来表达，但老师通常没有足够时间从 0 写故事、设计角色、分页、生成插图、排版和交付。

Kindleaf 的长期愿景是：

> 让每个幼儿园都能拥有自己的绘本内容资产库，让每位老师都能低成本创作适合教学场景的绘本，让每个孩子都能收到更贴近自己经历、兴趣和成长目标的故事。

## 3. 产品定位

Kindleaf 不是泛化 AI 写作工具，也不是单纯的儿童故事阅读平台。

它的核心定位是：

- 面向幼儿园工作场景的绘本生产工具。
- 面向园所协作的内容资产管理系统。
- 面向儿童个体差异的定制绘本生成系统。
- 面向优质内容复用的绘本市场平台。

系统的主线不是“输入一句话，生成一个故事”，而是把绘本生产拆成老师可确认、可修改、可复用的流程：

```text
教学目标
-> 普通绘本
-> 可复用母本
-> 儿童定制副本
-> 导出分享
-> 园所沉淀
-> 市场复用
```

## 4. 要解决的核心问题

### 4.1 老师创作成本高

老师知道教学目标，也知道班级孩子需要什么，但从 0 写出一本结构完整、语言适龄、画面可生成的绘本成本很高。

Kindleaf 要解决的是：

- 把教学目标转成故事方案。
- 把故事方案转成角色、道具和分页内容。
- 让老师在关键节点确认方向，减少一次生成到底带来的返工。

### 4.2 定制绘本效率低

给单个孩子生成定制绘本时，如果每次都从 0 生成，会带来质量不稳定、成本高、耗时长的问题。

Kindleaf 的解决方式是：

- 先生成普通绘本作为稳定母本。
- 再选择儿童档案派生定制副本。
- 定制过程只改应该个性化的部分，保留母本的结构稳定性。

### 4.3 园所内容难沉淀

很多优秀绘本会散落在老师个人电脑、聊天记录或一次性活动中，园所无法持续复用。

Kindleaf 要让普通绘本成为园所资产：

- 园所空间内保存和管理绘本。
- 老师之间可以复用优秀母本。
- 管理员可以把优秀普通绘本投稿到市场。
- 定制绘本和儿童隐私内容默认不公开。

### 4.4 儿童资料和分享存在隐私风险

定制绘本会使用儿童昵称、兴趣、性格、老师观察等信息，必须有明确的数据边界。

Kindleaf 的基本原则是：

- 账号只是登录身份，空间才是数据归属边界。
- 普通绘本可以共享和投稿。
- 定制绘本默认私有。
- 分享、导出、投稿前必须明确权限和隐私范围。

## 5. 目标用户

### 园所老师

园所老师是第一优先级用户。

他们需要：

- 创建课堂共读、规则引导、活动导入和家园沟通绘本。
- 基于班级儿童资料生成定制绘本。
- 编辑文字、角色、分页和插图。
- 导出 PDF 或创建分享链接。
- 复用园所或市场里的优秀绘本母本。

### 园所管理员

园所管理员是第二优先级用户。

他们需要：

- 管理园所空间、老师成员和班级。
- 查看园所绘本生产情况。
- 推动优秀绘本沉淀为园所内容资产。
- 管理市场投稿、隐私确认和基础审核状态。

### 个人用户

个人用户包括家长、独立老师或普通创作者。

他们需要：

- 创建个人普通绘本。
- 创建孩子资料。
- 为自己的孩子生成定制绘本。
- 导出或分享绘本。

### 链接访问者

链接访问者不一定登录。

他们主要通过链接：

- 提交儿童资料。
- 查看分享绘本。
- 下载或导出被允许访问的绘本。

## 6. 第一阶段目标

第一阶段目标不是一次性做完整平台，而是完成一个可演示、可验证、可继续扩展的核心闭环。

第一阶段必须跑通三条主路径。

### 6.1 普通绘本生成闭环

老师可以从教学目标创建一本普通绘本：

```text
填写需求
-> 生成故事方案
-> 确认角色和道具
-> 生成分页图文
-> 编辑页面
-> 生成或重绘插图
-> 预览
-> 导出或分享
```

验收标准：

- 老师不需要先写完整故事，也能得到一本结构完整的普通绘本。
- 故事方案阶段可以提前确认方向。
- 角色、分页文字和插图描述可以修改。
- 绘本可以保存、继续编辑、导出和分享。

### 6.2 单个儿童定制绘本闭环

老师可以基于普通绘本和儿童档案生成定制副本：

```text
选择普通绘本
-> 选择孩子
-> 检查儿童档案
-> 选择定制强度
-> 确认定制方案
-> 生成独立副本
-> 编辑
-> 导出或分享
```

验收标准：

- 原普通绘本不会被覆盖。
- 定制绘本能说明使用了哪些儿童信息。
- 儿童资料不足时有补充提示。
- 定制副本可以继续编辑、导出和分享。

### 6.3 园所内容沉淀与市场复用闭环

园所可以管理普通绘本，并把优秀作品沉淀或投稿：

```text
老师创建普通绘本
-> 保存到园所空间
-> 园所成员复用
-> 管理员发起投稿
-> 隐私确认
-> 进入市场审核状态
```

验收标准：

- 个人空间和园所空间边界清楚。
- `school_admin` 和 `school_teacher` 权限不同。
- 园所普通绘本可以作为定制绘本母本。
- 投稿前必须经过隐私确认。
- 市场模板可以复制到个人空间或园所空间。

## 7. 阶段规划

### 阶段一：可演示闭环

目标：让前端页面接入后端 API，完整演示普通绘本、定制绘本、市场复制、投稿、分享和导出流程。

本阶段可以使用 mock 生成内容，不要求真实 AI、真实 PDF 和真实对象存储。

必须完成：

- 登录、当前用户和空间列表。
- 个人空间、园所空间和空间角色。
- 儿童档案 CRUD。
- 普通绘本创建、编辑、详情、图片任务 mock、导出任务 mock、分享链接。
- 基于普通绘本生成单个儿童定制副本。
- 市场模板浏览、详情和复制。
- 园所成员、班级、投稿和隐私确认。
- 家长资料提交链接和绘本分享链接。

### 阶段二：数据库与权限稳定

目标：让数据持久化、权限边界和状态机可靠。

必须完成：

- PostgreSQL + SeaORM 数据模型。
- Workspace 级权限校验。
- `personal_owner`、`school_admin`、`school_teacher` 的权限测试。
- 绘本、页面、角色、儿童、市场、投稿、分享、导出的状态机。
- 基础审计记录，为隐私和管理操作留痕。

### 阶段三：真实生成能力

目标：把 mock 生成替换为真实 AI 生产链路。

必须完成：

- AI 文本生成。
- 角色一致性和插图生成。
- 图片重绘任务。
- PDF 排版和导出。
- 异步任务队列、进度轮询、失败重试。
- 生成成本和额度记录。

### 阶段四：生产化与商业化

目标：把产品从可演示系统推进到可试点、可运营、可收费。

后续完善：

- 邮件或短信邀请。
- 对象存储和文件 CDN。
- 平台市场审核后台。
- 投稿脱敏和内容安全审核。
- 园所用量、额度、账单和套餐。
- 部署监控、备份、安全策略。

## 8. 非目标

第一阶段暂不追求：

- 完整家长账号体系。
- 全班批量定制绘本。
- 真实支付和订阅。
- 复杂 BI 报表。
- 平台运营完整后台。
- 多园所集团化管理。
- 真实 AI 生成质量达到最终商业交付水平。

这些能力可以在核心闭环验证成立后逐步加入。

## 9. 产品原则

### 先普通绘本，后定制绘本

普通绘本是稳定母本，定制绘本是从母本派生的个性化副本。这样可以提高效率、降低成本，并为后续全班批量定制打基础。

### 先确认方案，再生成完整内容

系统应先让老师确认故事方案，再确认角色和道具，最后生成分页和插图。关键节点可控，才能减少大返工。

### 空间决定数据归属

账号负责登录，空间负责数据边界。绘本、儿童资料、分享链接、导出记录、投稿都必须归属到明确空间。

### 市场服务复用，不替代创作

市场不是阅读平台，而是给老师提供更高质量的创作起点。模板被复制后，应回到当前空间继续编辑和派生。

### 隐私优先于传播

只要内容包含儿童资料，就必须默认私有。公开、投稿和分享必须经过明确授权或隐私确认。

## 10. 北极星指标

第一阶段建议用一个核心指标判断产品是否成立：

```text
每个园所每周成功生成并交付的绘本数量
```

“成功交付”指绘本至少完成以下任一动作：

- 导出 PDF。
- 创建分享链接。
- 被复制为定制绘本。
- 被园所投稿或复用。

辅助指标：

- 普通绘本从需求到可导出的完成率。
- 普通绘本派生定制绘本的比例。
- 每本普通绘本平均派生的定制副本数量。
- 儿童档案完整度。
- 市场模板复制次数。
- 园所投稿通过率。

## 11. 最终验收目标

当以下场景能稳定演示并通过自动化检查时，可以认为第一阶段项目目标完成：

1. 老师登录后进入园所空间，创建一本普通绘本，并完成编辑、图片生成、导出和分享。
2. 老师创建或选择儿童档案，基于普通绘本生成该孩子的定制绘本副本。
3. 园所管理员能邀请老师、管理班级、提交市场投稿，并完成隐私确认。
4. 用户能从市场复制模板到当前空间，并继续编辑为自己的绘本。
5. 非登录访问者可以通过链接提交儿童资料或查看分享绘本。
6. API 权限能阻止未登录用户、非空间成员和错误角色访问不该访问的数据。
7. 前端主要页面不依赖纯 mock 数据即可完成核心流程。
