# 图片生成稳定性方案

## 目标

在儿童定制读本项目中，图片生成稳定性是核心能力。这里的稳定性不是指单张图片偶尔生成得很好，而是指系统能够以可预测、可复用、可控成本的方式，持续生成风格一致、角色一致、道具一致、质量达标的整本插图。

这份文档重点回答两个问题：
- 如何提高图片生成稳定性
- 如何通过稳定性降低整体生成成本

## 为什么稳定性是核心

在这个项目里，图片不是装饰，而是产品成立的关键部分。用户并不只关心“能不能生成图”，而是关心下面这些体验是否持续成立：

- 同一本书里的主角是否看起来是同一个孩子
- 重复出现的关键道具是否看起来是同一个道具
- 不同页面的画风是否一致
- 角色的衣服、发型、配件是否频繁漂移
- 场景是否符合故事内容
- 输出是否适合儿童读本，而不是偶尔失控

如果稳定性不够，会直接带来这些问题：
- 老师需要频繁重试
- 单页返工率升高
- 整本读本无法一次性交付
- 运营和客服成本增加
- 模型调用次数增加
- 单本读本毛利迅速下降

所以这个项目真正要优化的不是“单次生成效果最好”，而是“平均每本书成功交付所需的最少调用次数”。

## 稳定性的定义

在本项目中，图片生成稳定性建议拆成 6 个维度：

### 1. 角色一致性

同一本书中，主角应保持：
- 相同年龄感
- 相同发型
- 相同服装主色
- 相同标志性配件
- 相近脸部特征和体型比例

### 2. 道具一致性

如果某些道具会在同一本书中重复出现，应保持：
- 相同形状
- 相同主色和辅色
- 相同材质感
- 相同大小感
- 相同关键识别特征

### 3. 风格一致性

整本书应保持：
- 相同插画风格
- 相同线条密度
- 相同配色倾向
- 相同背景复杂度

### 4. 场景可控性

每一页的画面都应符合指定场景，而不是模型自由发挥：
- 教室就是教室
- 操场就是操场
- 分享玩具就是分享动作，而不是无关动作

### 5. 输出质量稳定性

应尽量减少：
- 肢体畸形
- 多余手指
- 成人化五官
- 背景过度复杂
- 主角被其他角色抢戏

### 6. 生产成本稳定性

不是偶尔生成成功，而是平均每本书的图片调用成本可预测：
- 单页重试率低
- 整本废弃率低
- 每次生成流程固定

## 不稳定的根本原因

### 1. 输入不标准

老师输入往往是自然语言，描述不统一：
- 有的写“活泼”
- 有的写“很爱跑”
- 有的写“经常笑”

如果直接把这些原始描述喂给图片模型，模型会每次抓不同重点，角色表现就会漂移。

### 2. Prompt 结构不固定

如果每次生成图片都临时拼 prompt，会出现：
- 风格词顺序变化
- 角色特征表达不一致
- 道具特征表达不一致
- 负面约束遗漏
- 场景描述粒度不统一

这会直接导致波动。

### 3. 没有角色锚点

如果每一页都只靠文字重新描述主角，模型会默认每一页都重新创造一次角色。

这在“单张图生成”里问题不大，但在“整本书插图”里会造成严重不一致。

### 4. 没有道具锚点

如果某个道具会在整本书中多次出现，但每次都只用自然语言临时描述，模型很容易让它出现漂移：
- 颜色变化
- 形状变化
- 大小变化
- 从玩具变成真实动物

这会削弱孩子和家长对故事连续性的感知。

### 5. 自由度过高

如果让老师任意选风格、任意改描述、任意改画法，模型空间会急剧扩大，稳定性会迅速下降。

### 6. 一步生成整本

如果试图一次性生成整本图文，问题会叠加：
- 任何一页失败都会影响整体交付
- 无法单页修复
- 难以定位是角色问题、道具问题、场景问题还是风格问题

### 7. 没有失败分层

有些失败其实不是模型能力不足，而是流程没分层：
- 角色卡错了
- 道具卡错了
- 页面描述错了
- 参考图没固定
- 重试逻辑不合理

如果所有失败都归结为“再生成一次”，成本只会越来越高。

## 核心原则

### 原则 1：不要直接生成页面图，先生成固定视觉定义

图片稳定性的第一步不是画图，而是先把“主角是谁”定义清楚。

建议先生成一份结构化角色卡，再以角色卡为基准生成后续插图。

如果书中有重复出现的关键道具，也应先生成结构化道具卡，而不是只固定主角。

### 原则 2：把变量和常量分开

必须区分：

常量：
- 主角年龄感
- 发型
- 服装主色
- 配件
- 高频重复道具
- 风格
- 比例

变量：
- 场景
- 动作
- 情绪
- 构图
- 配角

只有这样，系统才能在“每页都不同”的同时保持“主角始终相同、关键道具始终相同”。

### 原则 3：先锁角色和关键道具，再扩场景

不要先追求复杂场景和高表现力。先让系统稳定画出“同一个孩子”和“同一个关键道具”，再逐步提高背景和构图复杂度。

### 原则 4：少风格优先于多风格

MVP 不应提供大量风格。建议先只保留 2 到 3 种最容易控制的儿童绘本风格。

### 原则 5：宁可可识别，不要高度写实

这个项目不应追求“完全像真人孩子”。更稳的目标是：
- 发型能认出来
- 衣服颜色能认出来
- 配件能认出来
- 关键道具能认出来
- 整体气质能认出来

这样更容易控制一致性、审美和隐私风险。

## 推荐生成链路

建议把图片生成拆成 5 层。

### 第一层：儿童原始资料

老师输入：
- 姓名
- 年龄
- 照片
- 发型
- 常穿衣服
- 喜欢的颜色
- 兴趣标签

这一层允许非结构化，但不能直接送去画图。

### 第二层：角色卡规范化

先用文本模型把老师输入整理成标准角色卡。

示例：

```json
{
  "name": "小雨",
  "age_group": "5岁",
  "gender_expression": "女孩",
  "hair": "黑色短发，齐刘海",
  "skin_tone": "自然偏白",
  "outfit_top": "黄色卫衣",
  "outfit_bottom": "蓝色背带裤",
  "accessory": "小兔子发夹",
  "interest_elements": ["积木", "小兔子"],
  "style": "温暖扁平儿童绘本插画",
  "proportion": "幼儿比例，头稍大",
  "negative_rules": [
    "不要写实",
    "不要成人化五官",
    "不要复杂背景",
    "不要夸张光影",
    "不要额外主角"
  ]
}
```

这一层的价值是把模糊输入变成稳定输入。

### 第三层：角色标准图

系统先生成一张主角标准图，作为后续所有页面的角色锚点。

要求：
- 正面或 3/4 视角
- 姿态简单
- 背景尽量干净
- 能清楚看到发型、服装和配件

如果这一张不满意，先修这一张，而不是继续生成整本。

### 第四层：道具卡规范化

如果有重复出现的关键道具，应为它们建立结构化道具卡。

示例：

```json
{
  "prop_id": "prop_001",
  "name": "小兔子玩偶",
  "shape": "圆头长耳朵",
  "primary_color": "奶白色",
  "secondary_color": "浅粉色耳朵内侧",
  "material_style": "毛绒玩偶",
  "size": "适合幼儿抱在怀里",
  "visual_must_keep": [
    "长耳朵",
    "奶白色",
    "粉色耳朵内侧"
  ],
  "negative_rules": [
    "不要变成真实兔子",
    "不要变成卡通人物",
    "不要频繁改变颜色"
  ]
}
```

并不是所有道具都要建卡，只固定会反复出现、且会影响识别感的高频道具。

### 第五层：分页插图生成

后续每一页图片都由以下输入组成：
- 角色卡
- 角色标准图
- 道具卡列表
- 当前页场景描述
- 当前页构图描述
- 当前页动作描述
- 固定风格模板

这样每页变的是场景，不变的是角色和关键道具。

## 哪些道具需要固定

满足任一条件，就建议固定成 `prop_profile`：
- 同一本道具出现 2 次以上
- 道具和孩子身份强相关
- 道具承担情节推进作用
- 家长或老师会注意它是否一致

典型例子：
- 小书包
- 小兔子玩偶
- 恐龙玩具
- 水壶
- 红色雨伞
- 画笔盒

## Prompt 设计策略

### 1. 固定 Prompt 模板

不要让业务层每次动态自由拼装 prompt。应该使用固定模板，只替换少量变量。

建议分为五段：
- 角色固定设定
- 道具固定设定
- 风格固定设定
- 当前页面变量
- 负面约束

示例结构：

```text
你要绘制一张儿童绘本插画。

固定角色设定：
- 5岁女孩
- 黑色短发，齐刘海
- 黄色卫衣，蓝色背带裤
- 小兔子发夹
- 幼儿比例，头稍大

固定道具设定：
- 道具：小兔子玩偶
- 形状：圆头长耳朵
- 主色：奶白色
- 辅色：浅粉色耳朵内侧
- 材质：毛绒玩偶
- 一致性要求：必须与前序页面中相同道具保持一致

固定风格设定：
- 温暖扁平儿童绘本风
- 柔和配色
- 简洁背景
- 清晰表情

当前页面设定：
- 场景：幼儿园教室
- 动作：把积木递给同学
- 情绪：开心，主动分享
- 构图：中景，主角居中偏左

负面要求：
- 不要写实
- 不要成人五官
- 不要复杂光影
- 不要背景过满
- 不要出现第二个同样抢眼的主角
```

### 2. 固定字段顺序

Prompt 字段顺序不要频繁变化。字段顺序稳定，模型关注重点通常也更稳定。

### 3. 负面约束模板化

负面要求不能临时想一条写一条，应该按风格预置，也应该为高频道具预置。

### 4. 场景描述不要过长

场景描述过长会稀释主角和道具特征。每页描述只保留必要信息：
- 地点
- 动作
- 情绪
- 构图
- 关键背景元素

## 如何利用参考图提高稳定性

如果模型支持参考图、角色参考或图像条件输入，应优先使用。

推荐做法：
- 先生成主角标准图
- 如有高频道具，也保留道具参考图或固定道具描述
- 每页生成时都传入主角标准图
- 将“场景变化”作为文本变量

这样做的价值：
- 大幅降低角色漂移
- 降低关键道具漂移
- 减少只靠文字反复描述角色和道具的 token 成本
- 提高单页首次成功率

如果模型不支持强参考，也应保留标准图作为人工审核基准和重试依据。

## 如何降低成本

稳定性和成本并不是两件事，稳定性越高，成本通常越低。

### 1. 降低重试率

最直接的成本优化来自减少无效重试。

要减少重试率，应优先做：
- 角色卡规范化
- 道具卡规范化
- 固定 Prompt 模板
- 固定画风
- 角色标准图
- 单页重试而不是整本重做

### 2. 先低成本确认角色，再批量出页

不要在角色未确认时就生成整本。

更省成本的流程是：
1. 先生成角色卡
2. 先生成 1 张角色标准图
3. 人工确认角色可用
4. 再批量生成分页插图

这样可以避免整本图片因角色方向错误而全部报废。

### 3. 单页修复，不整本重跑

整本读本的失败通常不是整本都错，而是个别页面不稳定。

所以系统必须支持：
- 单页重绘
- 单页独立 prompt 调整
- 单页独立审核

### 4. 限制风格数量

风格越多：
- 调参越多
- 质检越难
- 重试逻辑越复杂
- 图片资产复用率越低

MVP 保留 2 到 3 种风格，能显著减少维护成本。

### 5. 建立失败分类

不要把所有失败都记成“生成失败”。建议至少分成：
- 角色不一致
- 道具不一致
- 构图不符合
- 场景不符合
- 质量异常
- 不适合儿童

只有这样，后续优化才知道该改哪一层。

## 质量控制机制

### 1. 生成前校验

在进入图片模型前先检查：
- 是否有完整角色卡
- 是否存在需要固定但未建卡的重复道具
- 是否选定固定风格
- 是否有场景描述
- 是否有标准角色图

### 2. 生成后自动质检

首期即使不做复杂视觉检测，也应有基础规则：
- 主角衣服主色是否偏离
- 重复道具的颜色和形状是否偏离
- 背景是否过度复杂
- 是否出现明显成人化特征
- 是否有多个抢主角角色

### 3. 人工快速复核

MVP 阶段不要假设全自动质检就够。可以保留一个低成本人工复核点，只判断：
- 像不像同一个孩子
- 重复道具是否保持一致
- 场景对不对
- 能不能交付

## 工程落地建议

建议把系统拆成四个能力层。

### 1. `character_builder`

职责：
- 接收老师输入的儿童资料
- 生成结构化角色卡
- 输出标准角色描述

### 2. `prop_builder`

职责：
- 识别哪些道具需要固定
- 生成结构化道具卡
- 输出关键道具描述

### 3. `scene_builder`

职责：
- 根据故事模板生成每页画面说明
- 明确每页场景、动作、情绪、构图

### 4. `image_generator`

职责：
- 接收角色卡、角色标准图、道具卡和页面说明
- 调用图片模型生成单页插图
- 保存 prompt 版本、角色版本和输出结果

## 必须记录的数据

每次图片生成都建议记录：
- `character_profile_version`
- `prop_profile_versions`
- `style_version`
- `prompt_template_version`
- `scene_version`
- `reference_image_id`
- `retry_count`
- `generation_cost`
- `review_result`

这些数据是后续稳定性优化的基础。没有记录，就无法知道成本为什么高、失败为什么发生。

## 推荐的 MVP 控制策略

如果当前目标是尽快上线一个可用版本，建议采用以下策略：

### 必做

- 固定 2 到 3 种儿童绘本风格
- 先生成角色卡
- 先生成标准角色图
- 对重复关键道具生成道具卡
- 每页使用固定 Prompt 模板
- 每页只改场景变量
- 支持单页重绘
- 记录失败类型和重试次数

### 不建议首期做

- 高自由度风格输入
- 完全自由文本转整本插图
- 真人高相似度还原
- 整本失败后整体重跑

## 判断稳定性是否改善的指标

建议至少跟踪以下指标：
- 单页首次通过率
- 单页平均重试次数
- 单本读本平均图片调用次数
- 单本读本图片生成成本
- 角色一致性人工通过率
- 道具一致性人工通过率
- 可直接交付率

如果这些指标持续改善，就说明稳定性方案有效，也意味着单位交付成本在下降。

## 结论

在这个项目中，图片生成稳定性本质上是一个系统设计问题，而不是单纯换一个更强模型的问题。

真正有效的做法是：
- 标准化输入
- 固定风格
- 固定 Prompt 结构
- 建立角色锚点和关键道具锚点
- 拆成多步生成
- 支持单页修复
- 记录失败原因

只有这样，才能把“偶尔生成得好”变成“稳定交付一本书”，并最终把模型调用成本压下来。
