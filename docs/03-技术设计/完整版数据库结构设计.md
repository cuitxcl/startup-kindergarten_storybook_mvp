# 幼儿园个性化内容平台完整版数据库结构设计

## Domain summary

首发版要支撑老师、园所、家长三方协同，以及儿童档案、角色设定、图片生成、内容生产、共享传播和成本治理。数据库必须从一开始就覆盖组织域、内容域、角色图像域和共享域，而不是只围绕单一本读本建模。

## Entities

- `schools`
  - 园所组织主体
- `teachers`
  - 教师账号
- `classrooms`
  - 班级组织单元
- `parents`
  - 家长账号
- `children`
  - 儿童档案
- `child_photos`
  - 儿童照片元数据
- `story_templates`
  - 系统内部使用的底层内容骨架
- `storybooks`
  - 一次具体生成内容
- `storybook_pages`
  - 内容逐页结构
- `character_profiles`
  - 儿童角色卡版本
- `parent_character_profiles`
  - 家长角色卡版本
- `reference_images`
  - 标准角色参考图
- `image_generation_tasks`
  - 图片生成异步任务
- `image_assets`
  - 生成后的图片资产
- `generation_cost_logs`
  - 生成成本记录

## Relationships

- 一个 `school` 包含多个 `classrooms` 和 `teachers`
- 一个 `classroom` 包含多个 `children`
- 一个 `parent` 可关联多个 `children`
- 一个 `child` 可拥有多版本 `character_profiles`
- 一个 `story_template` 可生成多本 `storybooks`
- 一本 `storybook` 包含多条 `storybook_pages`
- 一个 `storybook_page` 可关联多次 `image_generation_tasks`
- 一个 `image_generation_task` 最终可产出一个 `image_asset`

## Constraints and validation

- `children.name` 必填
- `children.status` 仅允许 `active` / `archived`
- `story_templates.status` 仅允许 `draft` / `active` / `archived`
- `story_templates.is_internal_only = true` 时，不直接暴露给老师端
- `storybooks.status` 建议限定为：
  - `draft`
  - `generating`
  - `ready`
  - `archived`
- `storybooks.content_type` 建议限定为：
  - `plain_storybook`
  - `custom_storybook`
  - `emotion_story`
  - `home_communication_card`
  - `growth_milestone_book`
- `storybooks.share_scope` 建议限定为：
  - `private`
  - `school`
  - `platform_review`
  - `platform_public`
- `storybooks.role_manifest_json` 必须至少记录主角、配角列表、角色关系和替换键
- `storybooks.derivation_type` 建议限定为：
  - `original`
  - `from_plain_storybook`
  - `from_custom_storybook`
  - `from_shared_library`
- `storybook_pages.page_role` 建议限定为：
  - `cover`
  - `story`
  - `closing`
- `image_generation_tasks.status` 建议限定为：
  - `queued`
  - `running`
  - `succeeded`
  - `failed`
- `storybook_pages(storybook_id, page_number)` 必须唯一
- 删除组织、老师、孩子时优先软删除，不应破坏历史生成内容

## Indexes and query patterns

建议索引：

- `children(primary_teacher_id, updated_at desc)`
- `children(classroom_id, updated_at desc)`
- `story_templates(content_type, theme, status)`
- `storybooks(teacher_id, updated_at desc)`
- `storybooks(child_id, created_at desc)`
- `storybooks(share_scope, updated_at desc)`
- `storybook_pages(storybook_id, page_number)`
- `image_generation_tasks(storybook_id, status)`
- `image_generation_tasks(storybook_page_id, status)`
- `generation_cost_logs(school_id, created_at desc)`

主要查询模式：

- 工作台加载老师最近内容和待处理提交
- 儿童列表按园所、班级、老师和提交状态筛选
- 老师端以前台成品案例为主进行筛选，后台仍需按模板骨架维护内容线与结构
- 编辑页一次取出一本内容、全部页面和图片状态
- 编辑页一次取出整本书的主角、配角和页面角色映射
- 共享库按共享范围和内容类型检索
- 园所后台按时间维度统计额度与成本

## Lifecycle rules

- `children`
  - 可由家长提交初稿，再由老师补充
- `character_profiles`
  - 每次显著设定变更应创建新版本
- `story_templates`
  - 作为底层骨架长期保留
  - 上线后优先归档而不是删除
- `storybooks`
  - 初始为 `draft`
  - 生成中为 `generating`
  - 完成后为 `ready`
  - 共享提交后进入审核或公开状态
  - 可作为前台案例再次被选择、复制和改编
  - 生成完成后应固化主角和配角信息，供后续角色替换使用
  - 普通绘本可作为母本，通过 `source_storybook_id` 派生出儿童定制绘本
- `storybook_pages`
  - 支持逐页编辑与锁定
  - 支持记录当前页涉及哪些角色
- `image_generation_tasks`
  - 允许失败后单页重试
- `generation_cost_logs`
  - 每次任务完成后落成本记录

## Migration notes

- 组织、内容、图片任务和成本表需要分批迁移，但模型边界应先一次性定清
- 角色卡、模板结构和 scene spec 使用 `jsonb`
- 前台案例展示依赖 `storybooks`，后台结构维护依赖 `story_templates`
- 角色替换优先基于 `role_manifest_json` 和已有页面资产做内容替换，不默认触发整本重生成
- 定制绘本优先视为从普通绘本母本派生，而不是独立重新生成
- 共享、导出、图片任务状态要统一枚举
- 如果现有迁移已覆盖部分表结构，应按完整版字段补齐而不是回退到简版

## Risks or alternative models

- 如果后续内容类型继续增加，`storybooks` 可能需要抽象成更通用的 `contents`
- 如果案例库越来越大，后续可能需要单独引入案例索引表
- 如果共享库审核规则复杂，可能需要单独的共享审核表
- 如果家长购买链路直接上线，后续还需要订单和支付域
