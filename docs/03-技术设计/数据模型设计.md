# 数据模型设计

## 目标

定义儿童定制读本项目的核心数据模型，用于统一：
- 产品字段理解
- 前后端接口契约
- 数据库存储设计
- 后续 ORM schema 和迁移实现

这份文档优先覆盖 MVP 和图片生成稳定性相关的数据结构。

## 设计原则

- 先满足 MVP 闭环
- 图片生成链路必须可追踪
- 角色设定必须版本化
- 单页生成必须可独立记录和重试
- 重要结构化字段优先于自由文本

## 核心实体总览

MVP 建议包含以下核心实体：

- `teachers`
- `schools`
- `classrooms`
- `parents`
- `children`
- `child_photos`
- `story_templates`
- `storybooks`
- `storybook_pages`
- `character_profiles`
- `parent_character_profiles`
- `reference_images`
- `image_generation_tasks`
- `image_assets`
- `generation_cost_logs`

## 实体关系

高层关系如下：

- 一个 `school` 可以有多个 `classrooms`
- 一个 `classroom` 可以有多个 `children`
- 一个 `teacher` 可以管理多个 `children`
- 一个 `parent` 可以关联多个 `children`
- 一个 `story_template` 可以生成多本 `storybooks`
- 一本 `storybook` 属于一个 `child`
- 一本 `storybook` 包含多页 `storybook_pages`
- 一个 `child` 可以有多个版本的 `character_profiles`
- 一个 `parent` 可以有多个版本的 `parent_character_profiles`
- 一个 `character_profile` 可以关联一个或多个 `reference_images`
- 一个 `storybook_page` 可以触发多个 `image_generation_tasks`
- 一个 `image_generation_task` 最终可能产出一个 `image_asset`

## 1. teachers

### 作用

存储教师账号信息。

### 建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| school_id | uuid nullable | 所属园所 |
| name | varchar | 教师姓名 |
| email | varchar nullable | 登录邮箱 |
| phone | varchar nullable | 登录手机号 |
| role | varchar | 角色，默认 `teacher` |
| status | varchar | `active` / `inactive` |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

## 4. children

### 作用

存储儿童基础档案。

### 建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| classroom_id | uuid nullable | 所属班级 |
| primary_teacher_id | uuid | 主要维护教师 |
| primary_parent_id | uuid nullable | 主要提交家长 |
| name | varchar | 姓名 |
| nickname | varchar nullable | 昵称 |
| age | int nullable | 年龄 |
| age_group | varchar nullable | 例如 `5岁` |
| gender_expression | varchar nullable | 展示用途 |
| personality_tags | jsonb | 个性标签数组 |
| interests | jsonb | 兴趣标签数组 |
| favorite_color | varchar nullable | 喜欢的颜色 |
| usual_outfit | varchar nullable | 常穿衣服描述 |
| notes | text nullable | 教师备注 |
| status | varchar | `active` / `archived` |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

### 说明

- `personality_tags` 建议用结构化数组，不直接写成长文本
- `notes` 用于保留教师自由补充信息


## 8. storybooks

### 作用

存储某个孩子的一本具体定制读本。

### 建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| child_id | uuid | 主角儿童 |
| teacher_id | uuid | 创建教师 |
| template_id | uuid nullable | 来源模板 |
| character_profile_id | uuid nullable | 生成时使用的角色卡 |
| character_profile_version | int nullable | 角色卡版本 |
| title | varchar | 读本标题 |
| theme | varchar | 主题 |
| teaching_goal | varchar nullable | 教学目标 |
| style_id | varchar nullable | 选中的插画风格 |
| reading_age_group | varchar nullable | 阅读年龄段 |
| status | varchar | `draft` / `generating` / `ready` / `archived` |
| export_status | varchar | `not_exported` / `exported` |
| share_status | varchar | `private` / `shared` |
| created_at | timestamp | 创建时间 |

- 一本 `storybook` 对应一次具体交付对象

## 10. character_profiles

### 作用

存储儿童规范化后的卡通人物形象角色卡。

### 建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| child_id | uuid | 所属儿童 |
| version | int | 版本号 |
| name | varchar | 角色名 |
| nickname | varchar nullable | 昵称 |
| age_group | varchar | 年龄层 |
| gender_expression | varchar nullable | 展示表达 |
| hair | varchar | 发型 |
| skin_tone | varchar nullable | 肤色 |
| face_shape | varchar nullable | 脸型 |
| body_proportion | varchar | 身体比例 |
| outfit_top | varchar nullable | 上装 |
| outfit_bottom | varchar nullable | 下装 |
| shoe | varchar nullable | 鞋子 |
| accessory | varchar nullable | 配件 |
| signature_colors | jsonb | 标志色数组 |
| interest_elements | jsonb | 兴趣元素数组 |
| visual_must_keep | jsonb | 必须保留的视觉特征 |
| negative_rules | jsonb | 负面约束 |
| source_photo_id | uuid nullable | 来源照片 |
| reference_image_id | uuid nullable | 标准角色图 |
| status | varchar | `draft` / `active` / `superseded` |
| created_by | uuid | 创建教师 |
| created_at | timestamp | 创建时间 |

### 版本规则

- 同一个 `child_id` 可有多个 `version`
- 新生成角色卡时不覆盖旧数据，直接新增版本
- `status = active` 表示当前推荐版本

## 13. image_generation_tasks

### 作用

记录图片生成任务，支持异步处理、重试和成本统计。

### 建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| task_type | varchar | `reference_image_generation` / `page_image_generation` / `storybook_image_generation` |
| parent_task_id | uuid nullable | 父任务 |
| storybook_id | uuid nullable | 所属读本 |
| page_id | uuid nullable | 所属页面 |
| character_profile_id | uuid | 角色卡 |
| character_profile_version | int | 角色卡版本 |
| parent_character_profile_ids | jsonb nullable | 家长角色卡列表 |
| reference_image_id | uuid nullable | 参考图 |
| style_id | varchar | 风格 |
| scene_spec_json | jsonb nullable | 页面场景 |
| prompt_template_version | varchar | Prompt 版本 |
| provider_name | varchar nullable | 服务商 |
| model_name | varchar nullable | 模型名 |
| status | varchar | `queued` / `running` / `succeeded` / `failed` / `needs_review` / `cancelled` |
| retry_count | int | 重试次数 |
| failure_reason | varchar nullable | 失败原因 |
| raw_prompt_text | text nullable | 最终拼装 Prompt |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |
| completed_at | timestamp nullable | 完成时间 |

### 说明

- 整本任务和单页任务建议都放在同一张表，通过 `task_type` 区分
- `parent_task_id` 用于将整本任务和单页子任务关联

## 推荐索引

至少建立以下索引：

- `children(classroom_id)`
- `children(primary_teacher_id)`
- `children(primary_parent_id)`
- `storybooks(child_id, created_at desc)`
- `storybook_pages(storybook_id, page_number)`
- `character_profiles(child_id, version desc)`
- `parent_character_profiles(parent_id, version desc)`
- `reference_images(character_profile_id, is_active)`
- `image_generation_tasks(storybook_id, status)`
- `image_generation_tasks(page_id, status)`
- `image_generation_tasks(character_profile_id, created_at desc)`
- `generation_cost_logs(task_id)`

## 推荐唯一约束

- `character_profiles(child_id, version)` 唯一
- `parent_character_profiles(parent_id, version)` 唯一
- `storybook_pages(storybook_id, page_number)` 唯一
- 可选：`reference_images(character_profile_id, is_active)` 通过业务约束保证单一激活项

## MVP 最小建表范围

如果首期资源有限，建议先建这几张表：

1. `teachers`
2. `parents`
3. `children`
4. `story_templates`
5. `storybooks`
6. `storybook_pages`
7. `character_profiles`
8. `parent_character_profiles`
9. `image_generation_tasks`
10. `image_assets`
11. `generation_cost_logs`

下面这些可以在后续补：

- `schools`
- `classrooms`
- `child_photos`
- `reference_images`

其中 `reference_images` 也可以首期并入 `image_assets`，但长期看单独拆表更清晰。

## 推荐实现顺序

### 阶段 1

- `children`
- `parents`
- `story_templates`
- `storybooks`
- `storybook_pages`

先把老师可见主流程跑通。

### 阶段 2

- `character_profiles`
- `parent_character_profiles`
- `image_generation_tasks`
- `image_assets`
- `generation_cost_logs`

把图片生成链路和成本追踪补齐。

### 阶段 3

- `schools`
- `classrooms`
- `child_photos`
- `reference_images`

把园所化和角色图管理能力补齐。

## 下一步建议

这份文档之后，最自然的下一步有两个：

1. 输出 `db-schema.sql` 或 ORM schema
2. 输出 `enum-and-validation-spec.md`，统一前后端字段枚举和校验规则

如果目标是直接开工开发，建议先做 schema。若目标是先把产品字段冻结，建议先做校验规范。
