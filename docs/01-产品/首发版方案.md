# 幼儿园个性化内容平台首发版方案

## Problem

幼儿园老师和园所需要一套能够稳定生产个性化教学内容、情绪引导内容、家园沟通内容和节日纪念内容的工具。单靠人工制作成本太高，通用 AI 工具又缺少儿童档案、角色一致性、成品案例复用、导出分享和园所管理能力，难以直接推广。

## Target user

主要用户：
- 幼儿园老师
- 园所管理员

次要用户：
- 家长

用户分工：
- 老师负责内容生产、编辑、导出和共享
- 园所负责账号、班级、额度和内容治理
- 家长负责提交儿童资料、接收内容和购买增值版本

## Goal

一步到位交付一套可推广的“幼儿园个性化内容平台”，覆盖儿童档案、家长资料提交、成品案例中心、图文生成、角色一致性、教师编辑、导出分享、班级管理、内容共享和园所级使用管理。

## Non-goals

- 不做面向所有学段的泛教育内容平台
- 不做完全开放式的通用 AI 创作工具
- 不以家庭 C 端独立创作为首要定位
- 不优先做音频朗读、拼音标注、阅读统计等外围能力

## User flow

1. 园所创建账号并配置老师、班级和额度。
2. 老师进入工作台，发起新内容或查看历史内容。
3. 家长通过邀请入口提交儿童基础档案、照片和家庭角色信息。
4. 老师补充儿童教学观察、兴趣标签和教学关注点。
5. 老师在内容中心选择内容线：
   - 个性化教学读本
   - 普通教学绘本
   - 情绪与规则引导故事
   - 家园沟通内容
   - 节日与成长纪念内容
6. 老师从成品案例中心挑选可直接套用的案例，配置孩子、画风、阅读年龄段和教学目标。
7. 如果老师选择普通绘本，系统先生成一本普通绘本母本；如果老师选择定制内容，系统也可基于普通绘本母本替换相关人物后生成定制绘本。
8. 系统生成角色卡、标准角色图和整本分页图文内容。
9. 老师在编辑页逐页改文案、重绘单页、锁定页面并预览整本效果。
10. 老师导出 PDF、电子翻页版、分享链接和二维码。
11. 老师可选择私有、园内共享或提交平台公开共享。
12. 其他老师可浏览共享内容、复制改编，并可基于普通绘本母本替换儿童角色派生出新的定制版本。

## Data implications

首发版需要完整支撑以下核心域：

- 组织与账号
  - `schools`
  - `teachers`
  - `classrooms`
  - `parents`
- 儿童与角色
  - `children`
  - `child_photos`
  - `character_profiles`
  - `parent_character_profiles`
  - `reference_images`
- 内容与底层骨架
  - `story_templates`
  - `storybooks`
  - `storybook_pages`
- 图片与成本
  - `image_generation_tasks`
  - `image_assets`
  - `generation_cost_logs`
- 共享与传播
  - 共享状态
  - 导出记录
  - 分享记录
  - 复制改编来源关系

## API implications

首发版 API 需要覆盖：

- 教师登录、园所和班级信息获取
- 家长资料提交和教师补充编辑
- 儿童档案、角色卡和参考图管理
- 成品案例列表、案例详情和分类筛选
- 底层模板骨架管理
- 读本生成、分页编辑、整本查询和复制改编
- 单页图片生成、整本图片生成、任务查询和重试
- PDF/电子版导出、分享链接、二维码
- 内容共享范围设置、脱敏、审核和共享库浏览
- 园所额度、生成成本和基础运营统计

## UI implications

首发版前端需要至少覆盖这些一级模块：

- 工作台
- 内容中心
- 儿童档案
- 成品案例中心
- 导出与分享
- 内容共享库
- 园所管理
- 数据与额度

同时需要提供：

- 家长资料提交入口
- 教师编辑器
- 图片重绘和任务状态反馈
- 共享设置与脱敏预览

## Functional requirements

- 支持教师登录和园所/班级上下文
- 支持家长提交儿童基础资料和照片
- 支持教师补充教学标签和教学关注点
- 支持儿童卡通主角和家长角色设定
- 支持成品案例分类、筛选、预览和选择
- 支持底层模板骨架供系统内部生成与维护使用
- 支持多内容线生成，不局限于单一定制读本
- 支持普通绘本生成，不强制绑定儿童档案
- 支持从普通绘本派生儿童定制绘本，通过替换人物生成新版本
- 支持角色一致性图片生成和单页重绘
- 支持逐页文案编辑、页面锁定、预览和保存
- 支持 PDF、电子版、分享链接、二维码和打印版
- 支持儿童历史内容、读本复制和替换主角生成
- 支持内容私有、园内共享和平台公开共享
- 支持共享内容的脱敏、审核和复制改编
- 支持园所级账号、班级、额度和成本管理

## Constraints

- 平台必须以幼儿园场景为核心，不做泛化设计
- 图片生成链路必须异步化、可追踪、可重试、可计费
- 内容共享默认私有，公开必须有明确脱敏规则
- 老师端流程不能因功能变多而失去主路径清晰度
- 园所级权限、儿童隐私和图片资产必须可控

## Phase 1 scope

首发版不是压缩版试探，而是面向推广的完整范围：

- 组织与账号体系
- 家长资料提交
- 儿童档案完整管理
- 多内容线成品案例中心
- 角色卡与参考图生成
- 整本图文生成与单页重绘
- 教师编辑器
- 导出、分享和打印
- 班级管理与历史记录
- 内容共享库
- 园所额度与成本管理

## Later phases

- 平台公开内容创作者激励
- 更细粒度运营报表
- 家长端购买流程增强
- 音频朗读和拼音标注
- 更高级审核和风控自动化

## Acceptance criteria

- 老师能独立完成从儿童资料准备到整本内容导出分享的完整流程。
- 家长能通过邀请入口完成儿童资料和照片提交。
- 系统能生成角色一致的多页插图，并支持单页独立重试。
- 老师能在同一平台内完成读本编辑、重绘、导出和共享设置。
- 园所管理员能管理老师、班级、额度和基础使用情况。
- 共享内容可按私有、园内共享、平台公开三种范围流转，并符合脱敏规则。
- 其他老师能基于共享内容进行复制改编，而不是只能查看。

## Open questions

- 园所管理和共享审核是否由同一后台承担，还是拆成教师端与管理端双视图？
- 家长增值购买在首发版是否直接接入支付，还是先以订单登记方式验证？
- 平台公开共享的审核机制由人工先做，还是要同时做规则引擎？

## Risks and assumptions

- 功能一步做全会显著增加交付复杂度，必须靠强约束的信息架构避免产品失焦。
- 组织、家长、老师三方链路同时上线，权限和状态流转会成为主要实现风险。
- 如果图片生成稳定性不够，推广时会直接伤害口碑。
- 如果共享脱敏做得不严格，隐私风险会高于普通教育工具。
- 如果前台只有成品案例、后台却没有稳定骨架，后续维护和批量优化会失控。
