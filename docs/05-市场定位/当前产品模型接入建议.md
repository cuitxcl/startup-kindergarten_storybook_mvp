# 当前产品模型接入建议

## 目标

基于当前儿童定制读本产品的实际阶段，给出一套可执行的模型接入建议，回答以下问题：
- 当前阶段应该优先接哪家厂商
- 文本和图片是否应该分开选型
- 如何降低接入风险
- 如何从 MVP 平滑过渡到小规模生产

## 当前产品阶段判断

从现有文档来看，当前产品仍处于从原型走向“可稳定交付”的阶段。当前最核心的问题不是功能够不够多，而是：

- 图片能否稳定生成
- 主角能否跨页一致
- 重复道具能否一致
- 老师能否在较低成本下完成单页修复
- 单本读本的平均生成成本能否被控制

因此，当前选型应该遵循一个原则：

先选最适合解决核心瓶颈的模型，而不是优先追求全平台统一。

## 选型原则

### 原则 1：文本和图片分开选

当前阶段不建议强行选择“同一家全包”。

原因：
- 图片稳定性是当前主问题
- 文本结构化输出是另一个问题
- 国内厂商在这两项上的优势并不完全重合

因此更合理的做法是：
- 文本选最适合结构化输出的
- 图片选最适合一致性生成的

### 原则 2：优先选能支持参考图和单页重绘的图片模型

你的产品不是一次性海报生成，而是绘本生产链路。能否支持：
- 参考图
- 多图输入
- 单页重试
- 连续多页生成

比“单张图好不好看”更重要。

### 原则 3：先做主链路，再保留备链路

不建议一开始就同时接 3 到 4 家厂商正式上线。

更合理的做法是：
- 先确定一条主图片链路
- 保留一条备选链路
- 后期再做 A/B 测试和降级策略

## 推荐方案

## 方案 A：当前最推荐

### 组合

- 文本模型：阿里 `Qwen`
- 图片模型：字节 `Seedream`

### 为什么这样选

`Qwen` 适合做：
- 故事生成
- 角色卡生成
- 道具卡生成
- 分页场景说明生成
- JSON 结构化输出

`Seedream` 适合做：
- 主角一致性生成
- 重复道具一致性生成
- 参考图驱动的分页插图
- 单页修复

### 适配当前产品的地方

这套组合最适合落地你目前文档里定义的链路：

1. 老师输入儿童资料
2. Qwen 生成 `character_profile`
3. Qwen 识别并生成 `prop_profile`
4. Qwen 生成 `scene_spec`
5. Seedream 根据角色卡、道具卡和场景描述逐页生成
6. 对失败页单独重试

### 优势

- 贴合当前核心问题
- 角色和道具一致性路线清晰
- 文本和图片各自选最强项
- 后续容易继续迭代 Prompt 和参考图策略

### 风险

- 需要接两家厂商
- 需要自己做统一的任务编排和成本统计

## 方案 B：稳妥的一体备选

### 组合

- 文本模型：腾讯混元
- 图片模型：腾讯混元生图

### 适合场景

如果你希望：
- 平台尽量统一
- 中文体验顺滑
- 图片修改过程更像对话式编辑

那腾讯是一条稳妥路线。

### 优势

- 同平台接入
- 中文语义和老师场景适配较好
- 多轮编辑体验好

### 风险

- 对后续批量生成的并发能力要尽早验证
- 在当前公开资料里，多参考图和组图思路不如字节路线清晰

## 方案 C：企业化长期方案

### 组合

- 文本模型：Qwen
- 图片模型：通义万相

### 适合场景

如果你的目标是：
- 后续卖给园所
- 统一使用阿里云体系
- 后端、鉴权、模型、存储尽量放在一个平台

这套方案很合适。

### 优势

- 平台整合度高
- 长期工程维护成本低
- 文本和图片都能在同一平台管理

### 风险

- 角色连续一致性需要你自己做更严格验证
- 当前未必是图片稳定性最优解

## 当前不建议的做法

### 1. 不建议一开始全量接入多家图片模型

原因：
- 成本高
- 任务编排复杂
- 排障困难
- 很难判断稳定性提升究竟来自模型还是来自流程

### 2. 不建议先追求完全单厂商统一

原因：
- 当前你的最大问题不是运维复杂度，而是图片稳定性
- 如果为了统一而牺牲核心效果，得不偿失

### 3. 不建议用低价模型直接承担主绘本链路

原因：
- 单价便宜不代表整本成本低
- 如果重试率高，最终成本反而更高

## 推荐接入路径

### 阶段 1：MVP 验证

目标：
- 验证角色一致性
- 验证道具一致性
- 验证单页重绘成本

建议：
- 文本接 `Qwen`
- 图片接 `Seedream`
- 固定 1 到 2 种画风
- 先做 2 到 3 个故事模板
- 先测试 3 个孩子、每人 1 本读本

### 阶段 2：小规模生产

目标：
- 验证真实成本
- 验证失败率
- 验证是否适合老师批量使用

建议：
- 保持主链路不变
- 新增腾讯图片链路做 A/B 对照
- 比较：
  - 单页首次成功率
  - 单本平均重试次数
  - 单本图片成本
  - 老师人工满意度

### 阶段 3：产品化决策

目标：
- 决定长期主供应商
- 决定是否需要双供应商架构

建议：
- 如果字节明显更稳，保留字节为主链路
- 如果腾讯在编辑体验上更优，可作为老师手动修图备链路
- 如果后续平台统一是首要目标，再考虑向阿里主栈靠拢

## 技术架构建议

建议采用“模型适配层”设计，不要在业务代码里直接写死某一家厂商。

### 推荐抽象层

- `text_model_adapter`
- `image_model_adapter`
- `image_edit_adapter`

### 推荐统一接口

- `generate_character_profile`
- `generate_prop_profiles`
- `generate_scene_specs`
- `generate_reference_image`
- `generate_page_image`
- `retry_page_image`

这样做的好处是：
- 后续替换厂商成本更低
- 更容易做 AB Test
- 更容易做失败降级

## 成本控制建议

当前阶段成本控制不应只看单次价格，而要看整本交付成本。

应重点统计：
- 单页首次成功率
- 单页平均重试次数
- 单本读本平均生成成本
- 单页修复成本
- 角色一致性人工通过率
- 道具一致性人工通过率

只有这些指标都建立后，模型选型才有真实依据。

## 最终建议

如果现在就要做决策，我建议：

### 当前主方案

- 文本：`Qwen`
- 图片：`Seedream`

### 当前备方案

- 图片备链路：`腾讯混元生图`

### 暂不作为主方案

- 智谱图片模型
- 百度图片模型

### 长期观察对象

- 阿里一体化方案

## 一句话结论

当前阶段最合理的不是“选一家最全的厂商”，而是“选最适合当前瓶颈的组合”。

对这个产品来说，当前最合理的组合是：

- 用 `Qwen` 负责结构化文本生成
- 用 `Seedream` 负责绘本图片稳定生成

等稳定性跑通，再决定是否做平台统一。
